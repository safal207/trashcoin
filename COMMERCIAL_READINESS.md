# Коммерческая оценка проекта trashcoin

## Короткий вывод
Сейчас проект **нельзя безопасно выводить в коммерцию как продукт с ИИ-классификацией**: классификация работает в режиме заглушки (случайный ответ), нет явной лицензии репозитория, отсутствуют базовые меры прод-безопасности и правовые документы для работы с персональными данными.

## Что уже есть
- Web-интерфейс на Flask с загрузкой изображений и выдачей результата.
- Базовый frontend (загрузка файла, предпросмотр, вызов `/classify`).
- Заготовка Telegram-бота.
- Инструкции деплоя на VPS / Render.

## Блокеры для коммерческого запуска

### 1) Продуктовая достоверность (критично)
- Модель не загружается: `load_trashnet_model()` возвращает `None`.
- При отсутствии модели `classify_trash()` возвращает случайный класс через `random.choice(...)`.
- Это означает, что пользователю показывается недостоверный результат, непригодный для платного сервиса.

### 2) Лицензирование и IP (критично)
- В репозитории отсутствует файл `LICENSE`.
- Без явной лицензии код по умолчанию защищён авторским правом, что затрудняет законное коммерческое использование третьими лицами и партнёрами.

### 3) Безопасность и эксплуатация (высокий риск)
- В `app.py` используется прямое имя загружаемого файла без `secure_filename`.
- Нет явных ограничений по размеру/типу файла на backend.
- В инструкции деплоя отсутствует обязательная настройка TLS (HTTPS) по умолчанию.

### 4) Юридическая часть и приватность (высокий риск)
- Проект собирает фото и геоданные, но нет:
  - Privacy Policy;
  - Terms of Service;
  - Data retention policy;
  - Cookie/consent-механики (если применимо).
- Для B2C/B2B-продаж это обычно обязательный минимум.

### 5) Надёжность ML-пайплайна (средний/высокий риск)
- Нет метрик качества модели (accuracy/F1/confusion matrix).
- Нет версионирования артефактов модели и мониторинга деградации.
- Нет MLOps/CI проверки качества перед релизом.

## Что нужно сделать, чтобы проект стал коммерчески готовым

### Фаза 1 (1–2 недели): «сделать честно рабочим»
1. Подключить реальную модель и убрать случайные ответы.
2. Добавить прозрачный флаг fallback-режима (если модель недоступна — явная ошибка, а не «угадайка»).
3. Ввести базовую валидацию загрузок (тип, размер, имя файла).
4. Добавить `LICENSE` (например, MIT/Apache-2.0 — по вашей бизнес-стратегии).

### Фаза 2 (2–4 недели): «готовность к продажам»
1. Подготовить Privacy Policy + Terms + политику хранения данных.
2. Включить HTTPS по умолчанию и секрет-менеджмент.
3. Добавить логирование, алерты, Sentry/аналог и health-check.
4. Сформировать baseline-метрики модели и SLA по качеству.

### Фаза 3: «масштабирование»
1. Оптимизировать latency/стоимость инференса.
2. Добавить A/B тесты и сбор обратной связи по ошибочным предсказаниям.
3. Описать тарифы и ограничения API (rate limits, fair use).

## Итог
- Как MVP/демо — проект хороший старт.
- Как коммерческий сервис «здесь и сейчас» — **пока рано**.
- После устранения критичных блокеров (реальная модель + лицензия + базовая безопасность + юридический пакет) можно выходить в пилотные продажи.
